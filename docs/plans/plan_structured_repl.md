# Plan: Structured REPL with watch / squash / KB + exec pane

Date: 2026-04-24
Status: Proposed — design phase
Inspiration: tmuxai (see `~/research/oss-agent-comparison-2026-04-22/repos/tmuxai/`)

## Problem

Today `tb agent repl <name>` is a line-oriented conversational REPL:

- Each user line runs the full plan/act loop with `tb_command` as the
  sole tool.
- The agent cannot *watch* a pane's state change over time.
- The agent cannot *attach* to a specific pane for its command output;
  it dispatches through the same `tb exec` shell-out every time.
- Long transcripts grow unbounded — no context compression.
- No way for the user to say "focus on the state of these three
  panes; ignore the rest."

tmuxai's UX solves this with three primitives:

1. **Observe / Watch / Prepare modes** — explicit stance on how
   proactively the agent should act.
2. **Chat pane + exec pane + read-only panes** — clear structural
   separation between interaction, execution, and context.
3. **Squashing** — compact the transcript into a summary once it gets
   long, preserving intent + salient state.
4. **Knowledge bases** — persistent per-session context loaded at turn
   start.

This plan adapts those primitives to tmux-browse's existing
architecture without forking away from it.

## Existing constraints to respect

- `tb agent` is a **single-tool loop** (`tb_command`). We don't want a
  second, parallel tool surface. Adding modes shouldn't require new
  adapters in `agent_providers.py`.
- Conversation persistence lives in `lib/agent_conversations.py` as
  append-only JSONL. Compaction needs to round-trip through that.
- The dashboard shows existing sessions as panes; a REPL "chat pane"
  already corresponds to a tmux session (`agent-repl-<name>`).
- Docker sandbox mode must still work — anything added here has to be
  orthogonal to the sandbox spec.

## Chosen model

A REPL **context** consisting of:

- Exactly one **exec target** — the tmux target the agent's `tb
  exec`/`tb type` calls default to.
- A list of **observed panes** — read-only targets the agent captures
  state from at turn start.
- A **mode** — one of `observe` (manual approval per tool call),
  `act` (auto-run tool calls), `watch` (periodic auto-turns triggered
  by observed-pane content change).
- A **knowledge set** — zero or more small text files preloaded into
  the system prompt.

Per-REPL state, stored alongside the conversation:
`~/.tmux-browse/agent-conversations/<name>/context.json`.

### New REPL slash-commands

All dispatched client-side in `tb agent repl`; the server sees them
as turn metadata or as plain-text edits to the system prompt.

| Command            | Effect                                                             |
|--------------------|--------------------------------------------------------------------|
| `/exec <target>`   | Set the exec target (`sandbox:` in Docker mode, otherwise any).    |
| `/watch <target>`  | Add an observed pane.                                              |
| `/unwatch <target>`| Remove one.                                                        |
| `/mode <m>`        | `observe` / `act` / `watch`.                                       |
| `/tick <sec>`      | Watch-mode poll interval (default 10 s, min 5 s).                  |
| `/squash`          | Summarize the transcript so far; replace turns with the summary.   |
| `/kb add <path>`   | Attach a file (up to 32 KiB) to the system prompt.                 |
| `/kb rm <name>`    | Detach.                                                            |
| `/kb ls`           | List attached KB files and their sizes.                            |
| `/context`         | Print the current context JSON.                                    |

All existing commands (`/history`, `/clear`, `/new`, `/fork`) continue
to work.

### Mode semantics

- **observe** (default; matches today): agent produces a plan;
  user confirms with Enter before any `tb_command` executes.
- **act**: tool calls execute immediately, no per-call confirmation.
- **watch**: between user turns, if any observed pane's
  `session_logs.idle_seconds` dips below the watch tick (indicating
  fresh output), re-run the agent with an automatic turn prompt
  like "context changed, decide next step or stay idle".

Watch mode is the most delicate — it's how an agent can drive a
long-running build by just starting it and waiting. Implementation
guard-rails:

- Watch-mode turns count against the existing `max_steps` + per-run
  token budget.
- A watch-mode run that ends with `"final"` of `"idle"` stops the
  next tick until observed-pane content changes again.
- `/mode observe` halts watch immediately.

### Squashing

`/squash` replaces all but the last turn with a single
`{role: "system", content: "[summary]"}` turn generated by calling
the provider with a dedicated compaction prompt:

> Compact this conversation into a short summary (≤ 300 tokens) that
> preserves: the user's objective, decisions already made, and the
> current state of each observed pane. Drop tool transcripts and
> intermediate reasoning.

The original conversation JSONL is **not rewritten** — the summary is
appended as a `compaction` record with a back-reference to the range
it summarized, and subsequent turns read from the summary onward.
This keeps the audit trail intact (critical for debugging when
compaction drops something important).

### Knowledge bases

A small disk-backed prompt prefix: `~/.tmux-browse/agent-kb/<name>/*.md`.
`/kb add <path>` copies the file in; the runner reads every file in
that dir at turn start and prepends them to the system message under
a `## Knowledge base` heading. Size cap per REPL: 128 KiB total;
files above cap are skipped with a warning.

## Module layout

- `lib/agent_repl_context.py` — new. Load/save `context.json`, mutate
  observed panes + exec target, render the "context block" that
  decorates the system prompt.
- `lib/agent_kb.py` — new. KB file management per conversation.
- `lib/agent_runtime.py` — extend. REPL loop recognizes slash-commands
  and routes them to the modules above instead of sending them to the
  provider.
- `lib/agent_runner.py` — extend. Accepts `context: ReplContext`
  parameter; prepends context block + KB to system prompt; uses the
  exec-target as the default `tb_command` target.
- `lib/agent_watch.py` — new. Polls observed panes via
  `session_logs.activity_ts`; triggers auto-turns when the hash shifts.
  Piggybacks on the existing `pollIdleOnly` infrastructure.

## Dashboard surface

This plan is CLI-first. Dashboard integration is a **separate
follow-up**; the CLI lands the semantics. Minimal dashboard hooks:

- `/api/agent-conversation/context` GET/POST — read or set a REPL's
  context from the dashboard.
- New "Context" tab inside each agent card: shows current exec target,
  observed panes, mode, KB files. Full editor is CLI-only for now.

## Non-goals

- **No full tmuxai UI port.** We're not forking into a pane-tiling REPL.
  The chat pane is still a ttyd in the dashboard; the exec pane is
  still whatever tmux session the user points at.
- **No multi-agent conversation.** One REPL, one agent, as today.
- **No persistent watch between REPL sessions.** Watch stops when the
  user exits the REPL. (Could be a conductor-level rule later.)
- **No provider-side tool-use schema.** Slash-commands are client-side
  only; the model never sees them directly.

## Tests

`tests/test_agent_repl_context.py`:

- Adding / removing observed panes round-trips through disk.
- Context block renders in the expected shape.

`tests/test_agent_kb.py`:

- `/kb add` respects the 128 KiB cap.
- Files over cap are skipped with a warning, not silently truncated.

`tests/test_agent_watch.py`:

- Observed pane hash change triggers one auto-turn.
- Unchanged pane does not.
- Mode transition to `observe` halts pending auto-turns.

Extend `tests/test_agent_runner.py` to cover the exec-target default
and KB prefix being present in the system prompt.

## Patch order

1. `lib/agent_repl_context.py` + tests.
2. `lib/agent_kb.py` + tests.
3. Wire slash-commands in `lib/agent_runtime.py` REPL.
4. Teach `agent_runner.run_agent` to consume context + KB.
5. `lib/agent_watch.py` + tests; integrate with `pollIdleOnly` path.
6. `lib/server.py` — minimal `/api/agent-conversation/context` endpoint.
7. Docs: `docs/tb.md` REPL section rewrite + `docs/recipes.md`
   examples (watch a build, attach KB).

## Acceptance criteria

- `tb agent repl opus` + `/watch dashboard:` + `/mode watch` causes
  opus to fire an auto-turn whenever something new lands in the
  dashboard session, and to stop ticking when it ends a turn with an
  `"idle"` final.
- `/kb add notes.md` preloads `notes.md` into every subsequent turn;
  `/kb ls` shows it; `/kb rm notes.md` removes it.
- `/squash` shortens the transcript without losing intent; the
  original turns remain readable via `/history` with a
  `[compaction]` marker.
- Docker sandbox mode + watch mode compose: observed panes can still
  be host-side (the agent's `tb exec` goes through the sandbox target,
  but its reads of observed panes hit the host via the same REPL
  helper, not the sandboxed `tb` — this is one of the subtle points
  that the plan needs to nail down before shipping).

## Open design questions

1. **Watch mode scope.** Should observed-pane reads in watch mode
   bypass the Docker sandbox (since they're context, not actions),
   or respect it? Proposal: **context reads bypass the sandbox,
   actions never do.** Needs a named helper and careful testing.
2. **Squashing cost.** Compaction is an LLM call that spends budget;
   should it be gated by the per-run budget or free? Proposal:
   charge it; the user pays for what they ask for.
3. **Observed pane cardinality.** Should there be a limit? tmuxai
   has no hard cap; we probably want one (say 8) to keep prompts
   reasonable.

Estimated scope: 2-3 full focused days. Largest surface area of
the five proposals — would benefit from being split into phases
(context + KB first; watch + squash second) for incremental shipping.
